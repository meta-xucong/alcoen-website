# ALCOCHROM Website Reconstruction Development Guide

## 1. Objective

This project is not meant to be a loose visual imitation of a corporate website. The objective is to reconstruct a fully editable corporate site based on the captured front-end and back-end structure of the target site, while using our own implementation.

Core principles:

- Reconstruct the front-end information architecture as closely as possible
- Reconstruct the back-end data model as closely as possible
- Use our own implementation rather than copying the target system code
- Finish a stable 1:1 structural reconstruction first, then refine styling, content, and interactions

## 3. Front-End Site Framework

### 3.1 Required Page Tree

Based on the target site and the captured column structure, the reconstructed front end should include:

1. Home page `/`
2. About page `/about`
3. Product list `/products`
4. Product detail `/products/:slug`
5. News list `/news`
6. News detail `/news/:slug`
7. Download page `/downloads`
8. Contact page `/contact`
9. Guestbook page `/guestbook`

### 3.2 Primary Navigation

The primary navigation should follow this structure:

- ABOUT
- PRODUCT
- NEWS
- DOWNLOAD
- CONTACT

`PRODUCT` must support a second-level category structure.

### 3.3 Product Category Structure

Confirmed product categories:

- PRODUCT
- Pharmaceutical
- Column
- Silica
- Accessories

`PRODUCT` is the top-level product section, while the other four are child categories.

### 3.4 Home Page Module Order

The home page should be organized in this order:

1. Header
2. Hero slider
3. Company or brand introduction
4. Product category or featured-product entry area
5. Product showcase
6. News showcase
7. Contact or lead-capture section
8. Footer

### 3.5 Single Page Template Structure

ABOUT, DOWNLOAD, and CONTACT belong to the single-page content model and should share a base template supporting:

- Page title
- Subtitle
- Top hero image or banner
- Rich-text body
- SEO title, keywords, and description

### 3.6 List Template Structure

At minimum, two list templates are needed:

- Product list template
- News list template

The product list template should support:

- Grouping by child category
- Category filtering or section switching
- Product cover image
- Product title
- Summary
- Detail entry

The news list template should support:

- Publish date
- Title
- Summary
- Thumbnail
- Detail entry

### 3.7 Detail Template Structure

The product detail page should support:

- Title
- Subtitle
- Main image
- Multi-image gallery or slider
- Rich-text body
- Extended parameter fields
- SEO information

The news detail page should support:

- Title
- Publish date
- Author and source
- Thumbnail
- Rich-text body
- SEO information

## 5. Data-Layer Design Principles

### 5.1 Recommended Top-Level Structure

The current [site.json](E:\AI\www.alcochrom.cn\data\site.json) has already been moved toward the captured back-end structure. The top-level keys should remain:

- `site`
- `company`
- `contentSorts`
- `singles`
- `news`
- `products`
- `slides`
- `links`
- `meta`

### 5.2 Data-Layer Requirements

- The front end and admin must share a single source of truth
- The front end must not depend on hardcoded content
- After admin saves, the front end must read the latest content immediately
- If any module is missing fields, the server must auto-fill safe defaults to prevent rendering failures
- Sorting fields must be applied consistently
- Rich-text fields may render HTML, but basic safety handling is required

### 5.3 File and Media Asset Rules

Recommended conventions:

- Store uploaded files under [public/uploads](E:\AI\www.alcochrom.cn\public\uploads)
- Store web-accessible paths in the data model
- Support both local uploads and remote image URLs
- Use a unified upload workflow for hero images, product images, thumbnails, and attachments

## 7. Back-End Development Guidance

### 7.1 Recommended Admin Strategy

The admin should use a `form-first + JSON fallback` approach:

- Form-based editing for normal content management
- An advanced JSON editor for debugging, import/export, and bulk correction

### 7.2 Suggested Admin Page Layout

Admin dashboard:

- Content summary
- Quick links
- Recent updates

Admin content pages:

- Left-side module navigation
- Right-side list or form editor
- Upload and preview support
- Save feedback and error feedback

### 7.3 Admin Interaction Requirements

- Create, delete, and save actions must have clear feedback
- Successful uploads should immediately fill the related field
- Sorting changes should take effect immediately
- Field names should stay as close as possible to the captured admin field names
- Missing content, missing image, and missing SEO data should all have safe defaults

## 8. Implementation Phases

### Phase 1: Unify the Data Model

Goal:

- Make the server, front end, and admin all operate on the new `site.json` structure

Tasks:

- Refactor [server.js](E:\AI\www.alcochrom.cn\server.js) to read and normalize the new schema
- Add default-value completion for `site`, `company`, `contentSorts`, `singles`, `news`, `products`, `slides`, and `links`
- Unify slug handling, category lookup, sorting, and status filtering

Delivery standard:

- Main routes render stably under the new schema

### Phase 2: Rebuild the Front-End Skeleton

Goal:

- Align the front-end page tree, category relationships, and list/detail structures with the captured target site

Tasks:

- Rebuild the home page module layout
- Unify the ABOUT, DOWNLOAD, and CONTACT single-page template
- Rebuild the PRODUCT list page with category grouping
- Rebuild the NEWS list and detail pages
- Unify header, footer, and breadcrumbs

Delivery standard:

- Page structure is substantially aligned with the target site

### Phase 3: Rebuild the Admin

Goal:

- Make the admin edit content using the real captured content model

Tasks:

- Rewrite [admin.html](E:\AI\www.alcochrom.cn\public\admin.html)
- Rewrite [admin.js](E:\AI\www.alcochrom.cn\public\admin.js)
- Keep image upload and advanced JSON editing
- Add modular editors for site, company, categories, single pages, news, products, slides, and links

Delivery standard:

- A non-technical operator can maintain content without editing raw JSON

### Phase 4: Visual Refinement

Goal:

- Make the front end visually close to the target site

Tasks:

- Refactor [styles.css](E:\AI\www.alcochrom.cn\public\styles.css)
- Adjust layout proportions, card styles, typography, spacing, backgrounds, and button states
- Ensure both desktop and mobile layouts are usable

Delivery standard:

- High structural match and strong visual similarity

### Phase 5: Integration and Stability

Goal:

- Ensure the front end remains stable after admin changes

Tasks:

- Save-path validation
- Upload regression checks
- Missing-field tolerance
- Route 404 and empty-content fallback behavior
- Invalid form-input protection

Delivery standard:

- Smooth editing in admin and stable rendering on the front end

## 10. Recommended Next Steps

The next development steps should be executed in this exact order:

1. Refactor [server.js](E:\AI\www.alcochrom.cn\server.js) so it fully supports the new schema
2. Rebuild front-end rendering based on the new data structure
3. Rebuild the admin UI
4. Refine styling and run regression checks

Why this order matters:

- Once the data model is stable, front-end and admin rework drops significantly
- If styling is polished before the schema is stable, a large portion of the work will likely be redone
